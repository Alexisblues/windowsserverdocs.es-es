# <a name="optimizing-windows-10-version-1803-for-a-virtual-desktop-infrastructure-vdi-role"></a>Optimización de Windows 10, versión 1803, para un rol de infraestructura de Escritorio Virtual (VDI)

En este artículo le ayuda a elegir la configuración para Windows 10, versión 1803 (compilación 17134) que debe producir el mejor rendimiento en un entorno de infraestructura de escritorio virtualizado (VDI). Todos los valores de esta guía son *recomendaciones para considerarse* y están en ningún requisito de forma.

En un entorno VDI formas clave para optimizar el rendimiento de Windows 10 son minimizar la aplicación gráfica vuelve a dibujar, actividades en segundo plano que no tengan ninguna ventaja importante para el entorno de VDI, y reducir de forma general los procesos en ejecución al mínimo. Un objetivo secundario es reducir al mínimo el uso de espacio en disco en la imagen base. Con VDI implementaciones, la base posible más pequeño o el tamaño de la imagen "gold", puede reducir ligeramente el uso de memoria en el hipervisor, así como una pequeña reducción en las operaciones de red general necesario para entregar la imagen de escritorio al consumidor.

> [!NOTE]  
> Configuración recomendada aquí puede aplicarse a otra instalación de Windows 10, versión 1803, los de físicos u otros dispositivos virtuales incluidos. No hay recomendaciones de este tema deben afectar a la compatibilidad de Windows 10, versión 1803.

> [!TIP]  
> Una secuencia de comandos que implementa las optimizaciones que se describen en este tema, así como un archivo de exportación de GPO que se puede importar con **LGPO.exe**--está disponible en [TheVDIGuys](https://github.com//TheVDIGuys) en GitHub.

## <a name="vdi-optimization-principles"></a>Principios de optimización de VDI

Un entorno VDI presenta una sesión de escritorio completo, incluidas las aplicaciones, para un usuario de equipo a través de una red. Entornos de VDI usan normalmente una imagen de sistema operativo base, que, a continuación, se convierte en la base para los escritorios que posteriormente se presentan a los usuarios para el trabajo. Hay variaciones de implementaciones de VDI como "permanente", "no persistente" y "sesión de escritorio". El tipo persistente conserva los cambios realizados en el sistema operativo de escritorio VDI de una sesión a la siguiente. El tipo no persistentes no conservar los cambios realizados en el sistema operativo de escritorio VDI de una sesión a la siguiente. Para el usuario este escritorio es diferente del otro dispositivo físico o virtual, que se tiene acceso a través de una red.

La configuración de optimización tenga lugar en un dispositivo de referencia. Una máquina virtual es un lugar ideal para generar la imagen, ya que puede guardar el estado, realizar puntos de comprobación y se pueden realizar copias de seguridad y otras tareas útiles. Empiece instalando el sistema operativo predeterminado en la máquina virtual base y, a continuación, optimice la VM base para su uso VDI quitando aplicaciones innecesarias, instalar las actualizaciones de Windows, instalar otras actualizaciones, eliminando archivos temporales, aplicar la configuración, etcetera.

Hay otros tipos de VDI como servicios de escritorio remotos y persistentes (RDS). Una explicación más detallada sobre estas tecnologías está fuera del ámbito de este tema, que se centra en el Windows base de configuración de la imagen con referencia a otros factores del entorno como optimización del host.

### <a name="persistent-vdi"></a>VDI persistente

En el nivel básico, VDI persistente es una máquina virtual que se guarda el estado del sistema operativo entre reinicios. Otras capas de software de la solución VDI proporcionan el acceso fácil y transparente de los usuarios a sus máquinas virtuales asignadas, a menudo con una solución de inicio de sesión única.

Hay varias implementaciones diferentes de VDI persistente:

-   Convencionales de máquinas virtuales, donde la máquina virtual tiene su propio archivo de disco virtual, se inicia normalmente, guarda los cambios de una sesión a la siguiente y básicamente es simplemente una máquina virtual normal. La diferencia es cómo el usuario tiene acceso a esta máquina virtual. Es posible que haya un portal web automáticamente, el usuario inicia sesión que dirige al usuario a su uno o más asigna las máquinas virtuales de VDI.

-   Basada en imagen persistente de máquina virtual, con los discos virtuales personales. En este tipo de implementación hay una imagen base/gold en uno o más servidores de host. Se crea una máquina virtual, y uno o más discos virtuales se crean y se asigna a este disco para el almacenamiento persistente.

    -   Cuando se inicia la máquina virtual, una copia de la imagen base se lee en la memoria de la máquina virtual. Al mismo tiempo, un disco virtual persistente asignado a esa máquina virtual, con los cambios del sistema operativo anterior combinados mediante un proceso complejo.

    -   Cambios tales como escrituras de registro de eventos, las escrituras de registro, etc. se redirigen al disco virtual de lectura/escritura asignado a esa máquina virtual.

    -   En este caso, sistema operativo y el mantenimiento de la aplicación podrían funcionar con normalidad, mediante software de mantenimiento tradicionales, como Windows Server Update Services u otras tecnologías de administración.

### <a name="non-persistent-vdi"></a>VDI no persistentes

Cuando una implementación de VDI no persistentes se basa en una base o una imagen "gold", las optimizaciones se realizan principalmente en la imagen base y, a continuación, a través de la configuración local y las directivas locales.

Con VDI no persistentes y basadas en imágenes, la imagen base es de solo lectura. Cuando se inicia una VM VDI no persistentes, una copia de la imagen base se transmite a la máquina virtual. Actividad que se produce durante el inicio y, a partir de entonces hasta el siguiente reinicio se redirige a una ubicación temporal. Normalmente, los usuarios se proporcionan las ubicaciones de red para almacenar sus datos. En algunos casos, el perfil del usuario se combina con la máquina virtual estándar para proporcionar al usuario su configuración.

Un aspecto importante de VDI no persitent que se basa en una sola imagen de mantenimiento. Actualizaciones del sistema operativo se entregan normalmente una vez al mes.
Con VDI basada en imágenes, hay un conjunto de procesos que se va a realizar con el fin de obtener las actualizaciones de la imagen:

-   En un host determinado, todas las máquinas virtuales en ese host que se derivan de la imagen base debe ser apagar o desactiva. Esto significa que los usuarios se redirigen a otras máquinas virtuales.

-   La imagen base, a continuación, se abre y se inicia. A continuación, se realizan todas las actividades de mantenimiento, como actualizaciones del sistema operativo, las actualizaciones. NET, las actualizaciones de aplicaciones, etcetera.

-   En este momento, se aplican las nuevas opciones que deben aplicarse.

-   Otras tareas de mantenimiento se realizan en este momento.

-   La imagen base, a continuación, se apaga.

-   La imagen base está sellada y establezca para volver al entorno de producción.

-   Los usuarios pueden volver a iniciar sesión.

> [!NOTE]  
> Windows 10 realiza un conjunto de tareas de mantenimiento automáticamente, de forma periódica. Hay una tarea programada que está configurada para ejecutarse en hora local de 3:00 A.M. cada día de forma predeterminada. Esta tarea programada realiza una lista de tareas, incluida la limpieza de Windows Update. Puede ver todas las categorías de mantenimiento que tienen lugar automáticamente con este comando de PowerShell:
>
>```powershell
>Get-ScheduledTask | ? {$_.Settings.MaintenanceSettings}
>```
>

Uno de los desafíos con VDI no persistentes es que cuando un usuario cierra la sesión, se descarta casi toda la actividad del sistema operativo. Es posible que se puede guardar el perfil del usuario y ni estado, pero la propia máquina virtual descarta casi todos los cambios realizados desde el último arranque. Por lo tanto, las optimizaciones destinadas a un equipo de Windows que se guarda el estado de una sesión en la siguiente no son aplicables.

Dependiendo de la arquitectura de VDI VM, las cosas like captura previa y SuperFetch no se servirá de Ayuda de una sesión al siguiente, como todas las optimizaciones se descartan en la máquina virtual se reinicie. La indización puede ser una pérdida parcial de recursos, como serían cualquier optimizaciones de disco como una desfragmentación tradicional.

### <a name="to-sysprep-or-not-sysprep"></a>Sysprep o no Sysprep

Se utiliza la herramienta Sysprep para preparar una imagen personalizada de Windows 10 para la duplicación. El proceso de Sysprep garantiza que el sistema operativo resultante es único correctamente para ejecutarse en producción.

Hay razones para y ante la ejecución de Sysprep. En el caso de VDI, es posible que desee la capacidad para personalizar el perfil de usuario predeterminado que se usará como la plantilla de perfil para posteriores usuarios inicie sesión con esta imagen. Puede que tenga aplicaciones que quiera instalada, pero también puede controlar la configuración por aplicación.

La alternativa es usar un estándar. ISO para instalar, posiblemente mediante un archivo de respuesta de instalación desatendida y una secuencia de tareas para instalar aplicaciones o quitar aplicaciones. También puede usar una secuencia de tareas para establecer la configuración de directiva local en la imagen, quizás mediante la [Local grupo Directiva objeto Utility (LGPO)](https://blogs.technet.microsoft.com/secguide/2016/01/21/lgpo-exe-local-group-policy-object-utility-v1-0/) herramienta.

Conectividad y control de tiempo son todo lo que al realizar la limpieza de la aplicación UWP. Si implementa su imagen base en un dispositivo sin conectividad de red, Windows 10 no puede conectarse a la Microsoft Store y descargar aplicaciones e intenta instalarlos mientras intenta desinstalarlas.

### <a name="clean-up-optional-features"></a>Limpiar las características opcionales

#### <a name="managing-optional-features-with-powershell"></a>Administrar características opcionales con PowerShell

 Para enumerar las características de Windows actualmente instaladas, ejecute este comando de PowerShell:

```powershell
Get-WindowsOptionalFeature -Online
```

Puede habilitar o deshabilitar una característica opcional de Windows específica como en este ejemplo:

```powershell
Enable-WindowsOptionalFeature -Online -FeatureName "DirectPlay" -All
```

Para obtener más información acerca de esto, consulte [Windows 10: Administrar características opcionales con PowerShell](https://social.technet.microsoft.com/wiki/contents/articles/39386.windows-10-managing-optional-features-with-powershell.aspx).

#### <a name="servicing-the-operating-system-and-apps"></a>El sistema operativo y aplicaciones de servicio

En algún momento durante el proceso de optimización de la imagen se deben aplicar las actualizaciones de Windows disponibles. Puede configurar Windows Update para instalar actualizaciones para otros productos de Microsoft, así como Windows. Para ello, abra **Windows configuración**, a continuación, seleccione **actualización y seguridad**y, a continuación, seleccione **opciones avanzadas**. Seleccione **ofrecer actualizaciones de otros productos de Microsoft al actualizar Windows** establecerlo en **en**.

Esto sería una buena configuración en caso de que va a instalar las aplicaciones de Microsoft, como Microsoft Office a la imagen base. De este modo Office está actualizado cuando la imagen se coloca en el servicio. También hay actualizaciones de .NET y ciertos componentes que no sean de Microsoft, como Adobe que tienen actualizaciones disponibles a través de Windows Update.

Una consideración importante para las máquinas virtuales de VDI no persistentes son actualizaciones de seguridad, incluidos los archivos de definición de software de seguridad. Estas actualizaciones se pueden liberar una vez o incluso más de una vez al día. Puede haber una manera de conservar estas actualizaciones, incluidas Windows Defender y componentes que no sean de Microsoft.

Para Windows Defender podría ser mejor que permita las actualizaciones que se produzca, incluso en VDI no persistentes. Las actualizaciones se van a aplicar a casi todas las sesiones de inicio de sesión, pero las actualizaciones son pequeñas y no deben ser un problema. Además, la máquina virtual no estará detrás de las actualizaciones porque se aplicará solo la más reciente disponible. El mismo puede ser true para los archivos de definición que no sean de Microsoft.

> [!NOTE]  
> Actualización de aplicaciones (aplicaciones UWP) mediante el Store Windows de Store. Actualización las versiones actuales de Office, como Office 365 a través de sus propios mecanismos de cuando se conecta directamente a Internet o a través de tecnologías de administración cuando no.

Consulte [Guía de implementación de Antivirus de Windows Defender en un entorno de infraestructura de escritorio virtual (VDI)](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-antivirus/deployment-vdi-windows-defender-antivirus) para obtener más información.

Las actualizaciones de firmas de Windows Defender pueden incluirse internamente en recursos compartidos de archivos y, en la práctica, tener esos recursos compartidos de archivos en los segmentos de red mismos o cerrados que las máquinas virtuales VDI.

### <a name="tuning-windows-10-network-performance-by-using-registry-settings"></a>Ajuste del rendimiento de red de Windows 10 mediante la configuración del registro

Esto es especialmente importante en entornos donde la VDI o equipo físico tiene una carga de trabajo es principalmente en función de red. La configuración en esta sección de rendimiento de sesgo a favor de las redes, mediante la configuración de almacenamiento en búfer y el almacenamiento en caché de las cosas adicionales como entradas de directorio y así sucesivamente.

Tenga en cuenta que algunas opciones en esta sección son *basada en registros sólo* y debe incorporarse en la imagen base antes de implementa la imagen para su uso en producción.

La siguiente configuración se documenta en el [directriz de ajuste de Windows Server 2016 rendimiento](https://docs.microsoft.com/windows-server/administration/performance-tuning/) information, publicado en Microsoft.com por el grupo de productos de Windows.

#### <a name="disablebandwidththrottling"></a>DisableBandwidthThrottling

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DisableBandwidthThrottling

Se aplica a Windows 10. El valor predeterminado es **0**. De manera predeterminada, el redirector de SMB limita el rendimiento en las conexiones de red de alta latencia, en algunos casos para evitar tiempos de espera relacionados con la red. Cuando este valor del registro **1** deshabilita esta limitación, lo que permite un mayor rendimiento de transferencia de archivos a través de conexiones de red de latencia alta, por lo que debe considerar esta opción.

#### <a name="fileinfocacheentriesmax"></a>FileInfoCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\FileInfoCacheEntriesMax

Se aplica a Windows 10. El valor predeterminado es **64**, con un intervalo válido de 1 a 65536. Este valor se usa para determinar la cantidad de metadatos de archivo que el cliente puede almacenar en caché. Aumentar el valor puede reducir el tráfico de red y aumentar el rendimiento cuando se tiene acceso a muchos archivos. Pruebe a aumentar este valor a **1024**.

#### <a name="directorycacheentriesmax"></a>DirectoryCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DirectoryCacheEntriesMax

Se aplica a Windows 10. El valor predeterminado es **16**, con un intervalo válido de 1 a 4096. Este valor se usa para determinar la cantidad de información de directorio que el cliente puede almacenar en caché. Aumentar el valor puede disminuir el tráfico y aumentar el rendimiento cuando se accede a directorios de gran tamaño. Considere la posibilidad de aumentar este valor a **1024**.

#### <a name="filenotfoundcacheentriesmax"></a>FileNotFoundCacheEntriesMax

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\FileNotFoundCacheEntriesMax

Se aplica a Windows 10. El valor predeterminado es **128**, con un intervalo válido de 1 a 65536. Este valor se usa para determinar la cantidad de información de nombres de archivos que el cliente puede almacenar en caché. Aumentar el valor puede reducir el tráfico de red y aumentar el rendimiento cuando se tiene acceso a muchos de los nombres de archivo. Considere la posibilidad de aumentar este valor a **2048**.

#### <a name="dormantfilelimit"></a>DormantFileLimit

HKLM\\System\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters\\DormantFileLimit

Se aplica a Windows 10. El valor predeterminado es **1023**. Este parámetro especifica el número máximo de archivos que se deben dejar abiertos en un recurso compartido una vez que la aplicación cerró el archivo. Si varios miles de clientes se conectan a servidores SMB, considere la posibilidad de reducir este valor para **256**.

Puede configurar muchas de estas opciones de SMB mediante el [conjunto SmbClientConfiguration](https://docs.microsoft.com/powershell/module/smbshare/set-smbclientconfiguration) y [conjunto SmbServerConfiguration](https://docs.microsoft.com/powershell/module/smbshare/set-smbserverconfiguration) cmdlets de Windows PowerShell. Puede configurar solo del registro mediante el uso de Windows PowerShell, como en el ejemplo siguiente:

`Set-ItemProperty -Path "HKLM:\\SYSTEM\\CurrentControlSet\\Services\\LanmanWorkstation\\Parameters" RequireSecuritySignature -Value 0 -Force`

### <a name="disk-cleanup-including-using-the-disk-cleanup-wizard"></a>Liberar espacio en disco (incluidos con el Asistente para liberar espacio en disco)

Liberar espacio en disco puede ser especialmente útil con implementaciones de VDI de imagen maestra. Después de la imagen maestra está preparada, actualizada y configurada, una de las tareas para realizar la última es liberar espacio en disco. El Asistente para liberar espacio en disco integrada en Windows puede ayudarle a limpiar a la mayoría de las posibles áreas de ahorro de espacio de disco.

> [!NOTE]  
> El Liberador de espacio en disco ya no se está desarrollando. Windows usará otros métodos para proporcionar funciones de limpieza de disco.



Éstas son sugerencias para varias tareas de limpieza de disco. Debe probar estos antes de implementar cualquiera de ellos:

1. Ejecute al Asistente para la limpieza de disco (con privilegios elevado) después de aplicar todas las actualizaciones. Incluyen las categorías **optimización de entrega** y **Windows Update limpieza**. Puede automatizar este proceso con **Cleanmgr.exe** con el **/sageset: 11** opción. Esta opción establece los valores del registro que se pueden usar posteriormente para automatizar la limpieza de disco, con todas las opciones disponibles en el Asistente para liberar espacio en disco.

   1.  En una prueba de la máquina virtual, desde una instalación limpia, ejecute **/sageset: 11 Cleanmgr.exe** revela que hay solo dos opciones de limpieza automáticos de disco habilitadas de forma predeterminada:

   - Archivos de programa descargados

   - Archivos temporales de Internet

   2. Si establece más opciones o todas las opciones, esas opciones se registran en el registro, según el valor de índice proporcionado en el comando anterior ( **/sageset: 11 Cleanmgr.exe**). En este ejemplo, usamos el valor *11* como nuestro índice para un procedimiento de limpieza de disco automatizado posteriores.

   3. Después de ejecutar **/sageset: 11 Cleanmgr.exe** , verá un número de categorías de las opciones del Liberador de espacio en disco. Puede seleccionar todas las opciones y, a continuación, seleccione **Aceptar**. Observará que el Asistente para liberar espacio en disco desaparezca. Sin embargo, la configuración seleccionada se guardan en el registro y se pueden invocar mediante la ejecución de **/sagerun: 11 Cleanmgr.exe**.

2. Limpiar el almacenamiento de instantáneas de volumen, si alguno está en uso. Para ello, ejecute los siguientes comandos en un símbolo del sistema con privilegios elevados:

   - **vssadmin list shadows**

   - **Vssadmin lista shadowstorage**

       Si el resultado de estos comandos se *encontró ningún elemento que satisface la consulta.* , entonces no hay ningún almacenamiento VSS en uso.

3. Limpieza de archivos temporal y los registros. Desde un símbolo del sistema con privilegios elevados, ejecute estos comandos:

   - **SUPR C:\\\*.tmp /s**

   - **SUPR C:\\Windows\\Temp\\.**

   - **SUPR % temp %\\.**

4. Eliminar los perfiles no utilizados en el sistema con este comando:

   **WMIC win32_UserProfile de ruta de acceso donde LocalPath = "c:\\\\usuarios\\\\\<usuario\>" eliminar**
