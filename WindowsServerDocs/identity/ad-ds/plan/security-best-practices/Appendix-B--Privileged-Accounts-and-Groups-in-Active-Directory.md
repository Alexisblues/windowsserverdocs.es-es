# <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Apéndice B: cuentas con privilegios y grupos de Active Directory

>Se aplica a: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Apéndice B: cuentas con privilegios y grupos de Active Directory  
"Con privilegios" cuentas y grupos de Active Directory son aquellos para que, privilegios, derechos eficaces se conceden permisos y que les permiten realizar prácticamente cualquier acción en Active Directory y en sistemas unidos a un dominio. Este apéndice se inicia con privilegios, derechos y permisos, seguido de información sobre las cuentas "privilegio más alto" y los grupos en Active Directory, es decir, los grupos y cuentas más eficaces.  

También se proporciona información sobre las cuentas integradas y de forma predeterminada y los grupos de Active Directory, además de sus derechos. Aunque se proporcionan recomendaciones de configuración específica para proteger los grupos y cuentas de privilegios más alto como apéndices independientes, este apéndice proporciona información general que ayuda a identificar los usuarios y grupos, que debes centrarte en proteger. Debe hacerlo porque se puedan aprovechar los atacantes comprometer la seguridad y destruir incluso la instalación de Active Directory.  

### <a name="rights-privileges-and-permissions-in-active-directory"></a>Derechos, privilegios y permisos en Active Directory  
Las diferencias entre los derechos, permisos y privilegios pueden ser confuso y contradictorio, incluso dentro de la documentación de Microsoft. Esta sección describe algunas de las características de cada uno, ya que se usan en este documento. Estas descripciones no deben considerarse autorizadas para otra documentación de Microsoft, ya que se pueden utilizar estos términos de manera diferente.  

#### <a name="rights-and-privileges"></a>Derechos y privilegios  
Derechos y privilegios son efectivamente las mismas capacidades de todo el sistema que se conceden a las entidades de seguridad, como los usuarios, servicios, equipos o grupos. En las interfaces que se suelen usados los profesionales de TI, por lo general se conocen como "derechos" o "derechos de usuario" y, a menudo se asignan los objetos de directiva de grupo. Captura de pantalla siguiente muestra algunos de los derechos de usuario más comunes que se pueden asignar a entidades de seguridad (que representa el GPO de controladores de dominio predeterminada de un dominio de Windows Server 2012). Algunos de estos derechos se aplican a Active Directory, como la **habilitar cuentas de usuario y el equipo de confianza para delegación** derecho de usuario, mientras que otros derechos se aplican en el sistema operativo de Windows, como **cambiar la hora del sistema**.

#### <a name="permissions"></a>Permisos  
Permisos son controles de acceso que se aplican a objetos protegibles como el sistema de archivos, registro, servicio y objetos de Active Directory. Cada objeto protegible tiene una lista de control de asociado de acceso (ACL), que contiene entradas de control de acceso (ACE) que concesión o denegación la capacidad de realizar varias operaciones en el objeto de entidades de seguridad (usuarios, servicios, equipos o grupos). Por ejemplo, las ACL para muchos de los objetos de Active Directory contienen las ACE que permiten a los usuarios autenticados a leer información general acerca de los objetos, pero no les otorgue la capacidad de leer la información confidencial o para cambiar los objetos.
A excepción de la cuenta Invitado integrada de cada dominio, cada entidad de seguridad que inicia sesión y se autentica en un controlador de dominio en un bosque de Active Directory o en un bosque de confianza tiene el identificador de seguridad de los usuarios autenticados (SID) agregado a su token de acceso de manera predeterminada. Por lo tanto, si un usuario, un servicio o una cuenta de equipo intenta leer generales propiedades en objetos de usuario en un dominio, la operación de lectura es correcta.  

Si se define una seguridad principal intenta obtener acceso a un objeto para que no ACE y que contienen un SID que está presente en el token de acceso de la entidad de seguridad, la entidad de seguridad no puede acceder al objeto. Además, si una ACE de ACL de un objeto contiene una entrada de denegación para un SID que coincida con el token de acceso del usuario, la "ACE de denegación de" por lo general, reemplazar un conflicto en "permitirá" ACE. Para obtener más información sobre el control de acceso en Windows, consulta [el Control de acceso](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) en el sitio Web MSDN.  

En este documento, permisos hace referencia a las funcionalidades que se concede o deniega a las entidades de seguridad en objetos protegibles. Siempre que haya un conflicto entre un derecho de usuario y un permiso, el derecho de usuario generalmente tiene prioridad. Por ejemplo, si un objeto de Active Directory se ha configurado con una ACL que impide que los administradores todo acceso de lectura y escritura a un objeto, un usuario que sea miembro del grupo de administradores del dominio podrá ver toda la información sobre el objeto. Sin embargo, porque el grupo de administradores tiene el usuario derecho "tomar posesión de archivos u otros objetos", el usuario puede simplemente tomar posesión del objeto en cuestión y luego vuelve a escribir la ACL del objeto para conceder el control total de los administradores del objeto.  

Es por este motivo que este documento te recomienda evitar mediante cuentas potente y grupos para la administración diaria, en lugar de intentar restringir las funcionalidades de las cuentas y grupos. No es eficaz posible impedir que un usuario determinado que tenga acceso a las credenciales eficaces de con las credenciales para acceder a cualquier recurso protegible.  

### <a name="built-in-privileged-accounts-and-groups"></a>Integrado privilegiadas cuentas y grupos  
Active Directory está pensada para facilitar la delegación de administración y el principio de privilegios mínimos en la asignación de derechos y permisos. Los usuarios "Normales" con cuentas en un dominio de Active Directory son, de manera predeterminada, puede leer gran parte de lo que se almacena en el directorio, pero pueden cambiar solo un conjunto muy limitado de datos en el directorio. Los usuarios que requieren privilegios adicionales pueden tener la pertenencia a grupos con privilegios que están integrados en el directorio para que se pueden realizar tareas específicas relacionadas con sus roles, pero no puede realizar tareas que no son relevantes para sus funciones.  

Dentro de Active Directory, hay tres grupos integrados que conforman los grupos de privilegios más altos en el directorio: el grupo de administradores de empresa (EA), el grupo de administradores de dominio (DA) y el grupo de administradores (BA) integrado.  

Un cuarto de grupo, el grupo de administradores de esquema (SA), que tenga privilegios que, si manipularse, pueden dañar o destruir todo un bosque de Active Directory, pero este grupo es mucho más en sus funcionalidades que la EA, DA y BA grupos restringidos.  

Además de estos cuatro grupos, hay una serie de integrado adicional y cuentas y grupos predeterminados en Active Directory, cada uno de ellos se concede derechos y permisos que permiten que las tareas administrativas específicas que se deben realizar. Aunque este apéndice no proporciona una explicación detallada de cada grupo predeterminado o integrados en Active Directory, proporciona una tabla de los grupos y cuentas que tengas más probable que vea las instalaciones.  

Por ejemplo, si instalas a Microsoft Exchange Server en un bosque de Active Directory, grupos y cuentas adicionales pueden crearse en los contenedores integrados y los usuarios en los dominios. Este apéndice describe solo los grupos y cuentas que se crean en los contenedores integrados y los usuarios en Active Directory, en función de las características y funciones nativas. Cuentas y grupos que se crean mediante la instalación del software de empresa no se incluyen.  

#### <a name="enterprise-admins"></a>Administradores de empresa  
El grupo de administradores de empresa (EA) se encuentra en el dominio raíz y de manera predeterminada, es un miembro del grupo integrado Administradores en cada dominio del bosque. La cuenta predefinida de administrador en el dominio raíz es el único miembro predeterminado del grupo de EA. EAs se otorgan derechos y permisos que puedan afectar a los cambios de todo el bosque. Estos son los cambios que afectan a todos los dominios del bosque, como agregar o quitar dominios, establecer relaciones de confianza de bosque o generar niveles funcionales de bosque. En un modelo de delegación correctamente diseñada e implementado, pertenencia EA es necesario solo cuando se construye primero el bosque o cuando se realicen cambios en todo el bosque, como establecer una relación de confianza de bosque de salida.  

El grupo EA se encuentra en el contenedor de los usuarios de dominio raíz del bosque de manera predeterminada, y es un grupo de seguridad universal, a menos que el dominio raíz se ejecuta en modo mixto de Windows 2000 Server, en cuyo caso el grupo es un grupo de seguridad global. Aunque algunos derechos se otorgan directamente al grupo EA, muchos de los derechos de este grupo en realidad son heredadas por el grupo EA porque es un miembro del grupo Administradores en cada dominio en el bosque. Administradores de empresa no tienen predeterminado derechos en estaciones de trabajo o servidores miembro.  

#### <a name="domain-admins"></a>Administradores de dominio  
Cada dominio en un bosque tiene su propio grupo de administradores de dominio (DA), que es un miembro del grupo de administradores (BA) integrado de ese dominio además de un miembro del grupo de administradores local en cada equipo que está unido al dominio. El único miembro predeterminado del grupo DA para un dominio es la cuenta predefinida de administrador para ese dominio.  

DAs son todopoderosos dentro de sus dominios, mientras que EAs tienen privilegios de todo el bosque. En un modelo de delegación correctamente diseñada e implementado, se deben DA pertenencia únicamente en escenarios "break vidrio", que son situaciones en que se necesita una cuenta con privilegios elevados en cada equipo en el dominio o cuándo se deben realizar algunos cambios ancho de dominio. Aunque los mecanismos de delegación de Active Directory nativos permitir la delegación en la medida en que es posible usar cuentas de DA solo en situaciones de emergencia, construir un modelo de delegación eficaz puede tardar mucho tiempo y muchas organizaciones usan las aplicaciones de terceros para acelerar el proceso.  

El grupo de DA es un grupo de seguridad global que se encuentra en el contenedor de los usuarios del dominio. Hay un grupo de DA para cada dominio en el bosque y el único miembro de forma predeterminada de un grupo de DA es la cuenta del dominio predefinida de administrador. Dado que se anida DA de un grupo en el grupo de BA del dominio y grupo de administradores locales de cada sistema Unidos a un dominio, DAs no solo tienen permisos concedidos específicamente a los administradores de dominio, pero también se heredan todos los derechos y permisos otorgados al grupo de administradores del dominio y grupo de administradores locales en todos los sistemas unidos al dominio.  

#### <a name="administrators"></a>Administradores  
Grupo integrado Administradores (BA) es un grupo local en el contenedor integradas de un dominio en el que están anidados DAs y EAs, y es este grupo que se concede muchos de los derechos directos y permisos en el directorio y en los controladores de dominio. Sin embargo, el grupo Administradores de un dominio no tienen privilegios en los servidores miembro o en estaciones de trabajo. La pertenencia a grupo de administradores local de equipos unidos a un dominio es donde se concesión privilegios local; y de los grupos que se tratan, DAs solo son miembros del grupo de administradores local de todos los Unidos al dominio los equipos de manera predeterminada.  

El grupo de administradores es un grupo de dominio local en el contenedor del dominio integrada. De manera predeterminada, cada BA grupo contiene la cuenta predefinida de administrador del dominio local, grupo de DA local del dominio y grupo de dominio raíz del bosque EA. Muchos de los derechos de usuario en Active Directory y en los controladores de dominio se otorgan específicamente para el grupo de administradores, no EAs o DAs. Grupo de BA de un dominio se concede permisos de control total en la mayoría de los objetos de directorio y puede tomar posesión de objetos de directorio. Aunque EA y DA los grupos se conceden determinados permisos específicos de un objeto en el bosque y dominios, gran parte de la potencia de grupos es realmente "heredado" de su pertenencia a grupos de BA.  

> [!NOTE]  
> Aunque se trata de las configuraciones predeterminadas de estos grupos con privilegios, un miembro de cualquiera de los tres grupos puede manipular el directorio para obtener la pertenencia a cualquiera de los otros grupos. En algunos casos, es algo trivial lograr, mientras que en otros es más difícil, pero desde la perspectiva de privilegios posibles, los tres grupos deben considerarse realmente el equivalente.  

#### <a name="schema-admins"></a>Administradores de esquema  
El grupo de administradores de esquema (SA) es un grupo universal en el dominio raíz del bosque y tiene solo cuenta predefinida de administrador de ese dominio como miembro de forma predeterminada, similar al grupo de EA. Aunque la pertenencia al grupo SA puede permitir que un atacante poner en peligro el esquema de Active Directory, que es el marco de todo el bosque de Active Directory, SAs tienen algunos derechos y permisos predeterminados más allá del esquema.  

Cuidadosamente debes administrar y supervisar la pertenencia al grupo de asociación de seguridad, pero en algunos casos, este grupo es "menos privilegiadas" que los tres más altos privilegiadas grupos descritos anteriormente, como el ámbito de sus privilegios es muy estrecho; es decir, SAs no tienen derechos administrativos en cualquier lugar que no sea el esquema.  

#### <a name="additional-built-in-and-default-groups-in-active-directory"></a>Integrados adicionales y grupos predeterminados en Active Directory  
Para facilitar la delegación de administración en el directorio, Active Directory viene con varios grupos integrados y predeterminado que se han concedido permisos y derechos específicos. Estos grupos se describen brevemente en la siguiente tabla.  

La siguiente tabla enumera los grupos integradas y de forma predeterminada en Active Directory. Existen dos conjuntos de grupos de forma predeterminada; Sin embargo, los grupos integrados se encuentran (de forma predeterminada) en el contenedor integrado en Active Directory, mientras se encuentran grupos predeterminados (de forma predeterminada) en el contenedor de los usuarios de Active Directory. Grupos en el contenedor integrado son todos los grupos de dominio Local, mientras que los grupos en el contenedor de los usuarios son una mezcla de grupos locales de dominio, Global y Universal, además de tres cuentas de usuario individual (Administrador, invitado y Krbtgt).  

Además de los grupos con privilegios más altos que se describió anteriormente en este apéndice, algunos grupos y cuentas integrada y predeterminado se otorgan privilegios elevados y deben también protegido y usarse únicamente en los hosts administrativos seguros. Estos grupos y cuentas pueden encontrarse en las filas sombreadas en la tabla B-1: integrado y cuentas en Active Directory y grupos predeterminados. Dado que algunos de estos grupos y cuentas se otorgan derechos y permisos que se pueden usar incorrectamente para poner en peligro Active Directory o controladores de dominio, se contemplados protecciones adicionales como se describe en [Apéndice C: protegido cuentas y los grupos de Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).
